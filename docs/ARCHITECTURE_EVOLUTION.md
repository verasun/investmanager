## 1. 架构演进历程

### 1.1 版本 0.1 - 单体应用 (2024-03)

**架构图:**
```
┌─────────────────────────────────────────────────────────┐
│                    单体应用                              │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐   │
│  │ API     │  │ 数据层  │  │ 分析层  │  │ 报告层  │   │
│  │ FastAPI │  │ PostgreSQL│ │ Pandas  │  │ Jinja2  │   │
│  └─────────┘  └─────────┘  └─────────┘  └─────────┘   │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐                │
│  │ Web UI  │  │ Redis   │  │ 邮件    │                │
│  │Streamlit│  │  Cache  │  │ SMTP    │                │
│  └─────────┘  └─────────┘  └─────────┘                │
└─────────────────────────────────────────────────────────┘
```

**特点:**
- 所有功能集成在一个应用中
- 依赖外部 PostgreSQL 和 Redis
- 部署复杂，需要多个容器

**痛点:**
- 部署依赖重（PostgreSQL + Redis）
- 资源占用高
- 启动慢，调试困难

### 1.2 版本 1.0 - 简化单体 (2024-03)

**架构图:**
```
┌─────────────────────────────────────────────────────────┐
│                   简化单体应用                          │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐   │
│  │ API     │  │ SQLite  │  │ LRU     │  │ OAuth2  │   │
│  │ FastAPI │  │ 本地存储 │  │ Cache   │  │ 邮件    │   │
│  └─────────┘  └─────────┘  └─────────┘  └─────────┘   │
│  ┌─────────────────────────────────────────────────┐   │
│  │              飞书机器人集成                      │   │
│  └─────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

**重大变更:**
- PostgreSQL → SQLite: 嵌入式数据库，零配置
- Redis → LRU Cache: 内存缓存，无需外部依赖
- SMTP密码 → OAuth2: 安全的邮件认证

**改进:**
- 单容器部署
- 启动时间 < 5秒
- 内存占用 < 200MB

### 1.4 版本 1.2 - 多进程架构 (2026-03-20)

**架构图:**
```
                           ┌─────────────┐
                           │   Feishu    │
                           │   Webhook   │
                           └──────┬──────┘
                                  │
                                  ▼
┌─────────────────────────────────────────────────────────────────┐
│                        GATEWAY (:8000)                          │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────────────┐│
│  │ Webhook     │  │ LLM Proxy   │  │ Capability Router       ││
│  │ Handler     │  │             │  │ (Mode-based routing)    ││
│  └─────────────┘  └─────────────┘  └─────────────────────────┘│
└───────────────────────────┬─────────────────────────────────────┘
                            │
        ┌───────────────────┼───────────────────┐
        │                   │                   │
        ▼                   ▼                   ▼
┌───────────────┐   ┌───────────────┐   ┌───────────────┐
│ LLM SERVICE   │   │  INVEST       │   │   CHAT        │
│   :8001       │   │   :8010       │   │   :8011       │
│               │   │               │   │               │
│ Alibaba       │   │ Stock         │   │ Personalized  │
│ Bailian       │   │ Analysis      │   │ Conversation  │
│ Qwen/Kimi     │   │ Backtest      │   │ Memory        │
└───────────────┘   └───────────────┘   └───────────────┘
                            │
                            ▼
                    ┌───────────────┐
                    │    DEV        │
                    │   :8012       │
                    │               │
                    │ Claude Code   │
                    │ Integration   │
                    └───────────────┘
```

**设计目标:**
- 关注点分离：每个服务专注单一职责
- 独立扩展：可单独扩展 LLM 或特定能力
- 故障隔离：单服务故障不影响整体
- 开发便利：可独立开发和测试

## 3. 当前架构详解

### 3.1 服务清单

| 服务 | 端口 | 职责 | 技术栈 |
|------|------|------|--------|
| Gateway | 8000 | 流量入口、路由、Webhook | FastAPI |
| LLM | 8001 | LLM API 代理 | FastAPI |
| Invest | 8010 | 投资分析能力 | FastAPI |
| Chat | 8011 | 对话能力+个人化 | FastAPI |
| Dev | 8012 | 开发模式+Claude Code | FastAPI |

### 3.2 数据流

```
用户消息 → Gateway → (模式判断) → 能力服务 → LLM → 响应
                │
                ├─ INVEST → Invest Service (8010)
                ├─ CHAT → Chat Service (8011)
                └─ DEV → Dev Service (8012)
```

### 3.3 服务间通信

```python
# Gateway → Capability Services
POST http://localhost:8010/process
POST http://localhost:8011/process
POST http://localhost:8012/process

# Capability Services → LLM Service
POST http://localhost:8001/chat
```

## 4. 设计决策记录

### 4.1 为什么选择多进程而非微服务？

**决策**: 使用多进程架构，而非独立的微服务

**理由**:
1. **部署简单**: 单机部署，无需 Kubernetes
2. **调试方便**: 共享代码库，日志集中
3. **资源高效**: 无网络序列化开销
4. **渐进演进**: 可平滑迁移到微服务

### 4.2 为什么独立 LLM 服务？

**决策**: 将 LLM 调用独立为服务

**理由**:
1. **API 限流**: 可独立控制 LLM 调用频率
2. **模型切换**: 无需重启其他服务
3. **成本追踪**: 独立监控 API 调用量
4. **缓存优化**: 可实现响应缓存

### 4.3 为什么保留 SQLite？

**决策**: 继续使用 SQLite 而非引入 PostgreSQL

**理由**:
1. **部署简单**: 零配置，无需 DBA
2. **性能足够**: WAL 模式支持并发读
3. **数据本地化**: 便于备份和迁移
4. **资源节约**: 无额外内存占用

### 4.4 DEV 模式设计

**决策**: DEV 模式通过 Claude Code CLI 集成

**理由**:
1. **开发效率**: 直接使用 Claude Code 能力
2. **上下文感知**: 工作目录配置灵活
3. **安全隔离**: 子进程执行，可控制权限
